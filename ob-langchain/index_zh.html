背景

为什么 AI 应用需要 OceanBase？

数据孤岛爆炸

向量、JSON、全文、GIS 分散在多个引擎，开发者被迫拼装 MySQL + ES + Pinecone 的"弗兰肯斯坦"架构。

新鲜度 vs 规模

传统向量库"最终一致性"导致 RAG 结果过时；OceanBase 严格 ACID 保证写入即可查。

胶水代码税

合并 MySQL + ES + Vector DB 结果需要 500+ 行 Python；OceanBase 一条 SQL 完成混合检索。

运维疲劳

多套集群、多种 API、多种故障模式——AI 应用不该这么复杂。

痛点：数据孤岛、一致性、胶水代码、运维复杂度。OceanBase 用一个引擎解决所有问题。
能力

OceanBase 向量能力一览

向量存储 & 索引

VECTOR 类型 + HNSW 索引，支持 L2 / Inner Product / Cosine 距离

全文检索

内置 FULLTEXT 索引，无需 Elasticsearch 外挂

混合搜索

DBMS_HYBRID_SEARCH 包：向量 + 全文 + 标量过滤，单查询完成

ACID 事务

标量、向量、全文索引原子更新，零不一致窗口

JSON & GIS

半结构化与空间数据统一存储，一个引擎覆盖全模态

MySQL 兼容

现有应用零改造迁移，保留完整 SQL 能力

OceanBase V4.3.3+ 原生支持向量类型和 HNSW 索引。核心优势：一个引擎搞定所有。
SDK

pyobvector — Python SDK 两种模式

Milvus 兼容模式

MilvusLikeClient 类，API 与 Milvus 对齐，迁移零成本。

Python
from pyobvector import MilvusLikeClient

client = MilvusLikeClient(
    uri="127.0.0.1:2881",
    user="root@test"
)
# 创建集合 + 向量索引
client.create_collection(
    collection_name="docs",
    schema=schema,
    index_params=idx_params,
)
SQLAlchemy 混合模式

ObVecClient 类，向量操作 + SQL 查询一体化，扩展 SQLAlchemy。

Python
from pyobvector import ObVecClient
from sqlalchemy import func

client = ObVecClient(
    uri="127.0.0.1:2881",
    user="root@test"
)
# ANN 搜索 + 标量过滤
res = client.ann_search(
    table_name="docs",
    vec_data=embedding,
    vec_column_name="embedding",
    distance_func=func.l2_distance,
    topk=5,
)
pyobvector 提供两种模式：Milvus 兼容（迁移友好）和 SQLAlchemy 混合（灵活查询）。安装：pip install pyobvector

RAG

端到端 RAG 管线

文档摄入

LangChain DocumentLoader → TextSplitter → Embeddings → OceanBaseVectorStore

混合检索

用户 Query → Embedding → 向量 + 全文混合搜索 → DBMS_HYBRID_SEARCH 加权融合

生成回答

Retriever → Context 拼接 → LLM (GPT-4 / Qwen / DeepSeek) → 结构化输出

评估迭代

LangSmith / RAGAS 评估 → 调整 chunk 大小、检索权重、reranker 策略

Python — RAG Chain
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough
from langchain_openai import ChatOpenAI

retriever = vs.as_retriever(search_kwargs={"k": 5})
llm = ChatOpenAI(model="gpt-4o")

rag_chain = (
    {"context": retriever | format_docs, "question": RunnablePassthrough()}
    | prompt
    | llm
    | StrOutputParser()
)

rag_chain.invoke("OceanBase 如何实现分布式一致性？")
完整的 RAG 管线：加载文档 → 分块 → 嵌入 → 存储 → 检索 → 生成。OceanBase 作为 VectorStore 替代 Pinecone/Weaviate。
混合搜索

DBMS_HYBRID_SEARCH 深度解析

01
标量过滤

WHERE 条件先缩小范围，从百万级到千级

02
粗排：向量 + 全文

HNSW 向量近邻 + FULLTEXT 关键词匹配，低成本并行召回

03
精排：Reranker

高精度重排序，可配置向量/全文权重比

SQL — 混合搜索
CALL DBMS_HYBRID_SEARCH(
  table_name   => 'knowledge_base',
  query_text   => '分布式事务一致性协议',
  vector_col   => 'embedding',
  fulltext_col => 'content',
  top_k        => 10,
  alpha        => 0.7   -- 向量权重 0.7, 全文权重 0.3
);
alpha 参数控制向量与全文的权重。0.7 表示 70% 语义相似度 + 30% 关键词匹配。可根据场景调整。

快速开始

5 分钟上手

Shell — Docker 启动 OceanBase
# 启动 OceanBase CE (向量能力需 V4.3.3+)
docker run -d --name ob \
  -p 2881:2881 \
  -e OB_MEMORY_LIMIT=6G \
  oceanbase/oceanbase-ce:4.3.5-lts
Shell — 安装 & 运行
pip install langchain-oceanbase pyobvector langchain-openai

# 或使用 seekdb 轻量版
docker run -d --name seekdb \
  -p 2881:2881 -p 2886:2886 \
  -v ./seekdb:/var/lib/oceanbase \
  oceanbase/seekdb:latest
Python — 最小 RAG 示例
from langchain_oceanbase import OceanBaseVectorStore
from langchain_openai import OpenAIEmbeddings, ChatOpenAI

vs = OceanBaseVectorStore(
    embedding=OpenAIEmbeddings(),
    connection_args={"host":"127.0.0.1","port":"2881",
                     "user":"root@test","database":"test"},
)
vs.add_texts(["OceanBase 是分布式关系数据库"])
print(vs.similarity_search("什么是 OceanBase?", k=1))
三步即可运行：1) Docker 启动 OB 2) pip install 3) Python 代码连接。seekdb 更轻量，1C2G 即可运行。
